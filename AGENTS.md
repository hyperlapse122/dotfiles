# AGENTS.md

Cross-platform dotfiles for **Windows + macOS + Fedora Linux**. Dotfiles are applied by [chezmoi](https://www.chezmoi.io/) invoked via mise-managed ephemeral binary — never installed globally.

> **Status**: fully bootstrapped and migrated. This repo is the single source of truth for every tracked dotfile — edit files HERE.

User-facing quickstart belongs in `README.md` (top-level). This file (`AGENTS.md`) is for agents only.

## Hard rules

### Branching — `main` only (overrides global Git Flow gate)

This repository commits directly to `main`. It does **not** use Git Flow branches (`feature/`, `bugfix/`, `hotfix/`, `refactor/`, `docs/`, `chore/`, `release/`) or any other topic-branch workflow. The branch-naming convention and pre-first-commit prefix gate in the global agent rules (`~/.config/opencode/AGENTS.md` / `~/.codex/AGENTS.md` / `~/.claude/CLAUDE.md`) **do not apply here** — this project-level rule overrides them per the precedence rule (project `AGENTS.md` wins on conflict).

- **Commit straight to `main`.** Do not create, rename to, or switch to a topic branch before committing.
- **Do not run the prefix gate** (`git branch --show-current` check, `git branch -m <prefix>/<slug>` rename) on this repo. It is a single-maintainer dotfiles repo; the Git Flow ceremony adds no value.
- **No PRs/MRs required.** Push commits to `main` directly. (If a PR is ever explicitly requested, open it from `main` against `main` of a fork, or as instructed — but the default is direct commit.)
- Everything else in the global rules still applies: Conventional Commits message format, no committing secrets, no destructive/bypass git operations without explicit confirmation, no git-config edits.

### chezmoi — mise-managed, NEVER install globally

- The **only** way chezmoi is invoked in this repo is `mise exec chezmoi@2.70.5 -- chezmoi ...`. mise fetches the pinned binary ephemerally, so every machine gets the exact same version regardless of local state. **Do not install chezmoi globally.** Forbidden, all of these:
  - `brew install chezmoi`, `dnf install chezmoi`, `apt install chezmoi`, `winget install chezmoi`, etc.
  - `go install github.com/twpayne/chezmoi/v2@latest`
  - using an ambient `chezmoi` already on `$PATH` — it may be a different version
- The single canonical invocation: `mise exec chezmoi@2.70.5 -- chezmoi apply --source "$REPO_ROOT/home"`. Both `install.sh` and `install.ps1` MUST call exactly this; nothing else may invoke chezmoi.
- **`.chezmoiroot = home`** — the chezmoi source root is the `home/` subdirectory of the repo. `.chezmoi.toml.tmpl` and `.chezmoiignore.tmpl` live inside `home/`, not at the repo root. Always pass `--source "$REPO_ROOT/home"` (or let chezmoi discover it via `.chezmoiroot`) rather than pointing at the repo root.
- **Source encoding** — chezmoi uses filename prefixes to encode metadata: `dot_` replaces a leading `.` in the target name; `private_` sets mode `0600`/`0700`; `executable_` sets the execute bit. Do not rename source files by hand without understanding the encoding — a wrong prefix silently changes the target path or permissions.
- **Managed-copy semantics** — chezmoi copies (or symlinks for `symlink_` entries) files from `home/` to `$HOME`. Edit the source under `home/`, then run `mise exec chezmoi@2.70.5 -- chezmoi apply` to push changes. Do **not** edit `$HOME` targets directly; chezmoi will overwrite them on the next apply.
- **Ownership boundary** — `.chezmoiignore.tmpl` separates chezmoi-managed content from orchestrator live-symlinks. Paths listed there (e.g. `agents/`, `.config/opencode/commands`, `.config/opencode/AGENTS.md`, `.claude/CLAUDE.md`, `.codex/AGENTS.md`, `.codex/prompts`, `.agents/`) are created by `scripts/bootstrap/link-repo-trees.sh`, not chezmoi. Do not add those paths to the chezmoi source tree.
- **Orchestrator ordering** — `install.sh` / `install.ps1` run steps in this order: verify mise → toolchain provision → remove-old-symlinks → glab skills → render prompts → `chezmoi apply` → setup-glab → cargo build → yarn build → `link-repo-trees.sh` → configure-codex → inject-secrets → Linux-only block (root-owned config, services). chezmoi apply runs before the live-symlink step so the target directories exist when `link-repo-trees.sh` creates symlinks into them.
- The bootstrap scripts MUST NOT install `mise`; users install `mise` themselves before running the scripts. The scripts may only verify that `mise` is available, then use `mise exec chezmoi@2.70.5 -- chezmoi ...`.
- **Platform gating** — use `.chezmoiignore.tmpl` with `.chezmoi.os` (`linux`, `darwin`, `windows`) to exclude OS-specific target paths on the wrong platform. Do not rely on shell `if:` directives or per-OS yaml files for chezmoi content.

### Script parity (HARD)

Every script ships in **both** forms or it is broken:

| Surface | Extension | Targets |
|---|---|---|
| POSIX | `.sh` (`#!/usr/bin/env bash`) | macOS + Linux |
| PowerShell | `.ps1` | Windows |

Adding `foo.sh` without `foo.ps1` is a regression. The two MUST behave equivalently for their target platforms; if a feature is impossible on one side, the script SHOULD exit with a clear error rather than silently no-op.

Exception: scripts that are inherently single-platform MAY skip parity — document the reason in a header comment in the script itself. Current exceptions: `scripts/linux/install-linux-system-config.sh` (writes to `/etc/`, Linux only), `scripts/linux/install-packages.sh` (uses `dnf`, Fedora only), `scripts/linux/config-kde.sh` (configures KDE Plasma 6, Linux only), `scripts/auth/auth-tailscale.sh` (runs Linux `tailscale up` with `sudo`), and `scripts/linux/setup-luks-tpm2-unlock.sh` (LUKS/TPM2/dracut, Linux only).

### Root-owned config (`/etc/...`)

Root-owned files live under `system/<os>/` mirroring their absolute install path:

```
system/linux/etc/NetworkManager/conf.d/wifi-powersave-off.conf
              -> /etc/NetworkManager/conf.d/wifi-powersave-off.conf
```

They are installed by `install.sh` calling `scripts/linux/install-linux-system-config.sh`, which runs:

```sh
sudo install -D -m <mode> system/<os>/<abs/path> /<abs/path>
```

Use `sudo install -D` (atomically sets mode and creates parents). Do **not** use `cp`+`chmod` (loses ownership/mode atomicity). These files are NOT managed by chezmoi — chezmoi only handles user-owned `$HOME` content.

The Linux installer discovers files with a recursive glob under `system/linux/etc/`; most files install at mode `0644`. Special cases are `system/linux/etc/sudoers.d/*` (mode `0440`, gated on `systemd-detect-virt --vm`, syntax-checked with `visudo -c -f`) and the ThinkPad `thinkpad_acpi` files under `modprobe.d/` + `modules-load.d/` (installed only when `dmidecode -t system` reports a ThinkPad, then loaded with `modprobe thinkpad_acpi`). Adding or removing a root-owned config file outside those gated paths should not require editing the install script; any other file that needs a non-default mode or a platform/host gate does.

NetworkManager unmanaged-device rules live as a single consolidated drop-in `system/linux/etc/NetworkManager/conf.d/99-unmanaged-devices.conf` that currently matches `podman*` (the rootless container interface, replacing the earlier `docker*` rule when the runtime switched), plus `veth*`, `tailscale*`, `vmnet*`, `lo`, `virbr*`, and a handful of CNI/overlay names. The historical reason for the consolidation still applies: the earlier per-purpose `80-lo.conf` / `90-unmanaged-vmware.conf` / `91-tailscale.conf` / `92-docker.conf` / `93-veth.conf` split files were rolled into a single `[keyfile]/unmanaged-devices=` key; one key per file is sufficient and easier to audit than five overlapping drop-ins. The deleted split files are listed in `REMOVED_ETC_PATHS` in `scripts/linux/install-linux-system-config.sh` so machines pulling the consolidation also drop the orphans. Do not consolidate further into a monolithic `NetworkManager.conf` — that file is intentionally absent from this repo (drop-ins under `conf.d/` are the supported integration point).

Current Linux root-owned config includes NetworkManager Wi-Fi power-save and unmanaged-device drop-ins, ThinkPad-only `thinkpad_acpi fan_control=1` module-load/modprobe drop-ins, keyd defaults, a libinput local override, `locale.conf`, Plymouth config, Logitech receiver and NuPhy Gem80 VIA/WebHID udev rules, IPv4/IPv6 forwarding via `sysctl.d/`, and a VM-only `sudoers.d/` drop-in granting `%wheel` password-less sudo. All install at mode `0644` except the `sudoers.d/` drop-in (mode `0440`, VM-only), with the ThinkPad files additionally gated by DMI detection — see `scripts/linux/install-linux-system-config.sh`. Containers run rootless under the user, with a user-scope `podman-prune.service` / `podman-prune.timer` pair under [`home/dot_config/systemd/user/`](home/dot_config/systemd/user/) that runs `podman system prune --force` + `podman volume prune --force` weekly (`Persistent=true`, `RandomizedDelaySec=1h`, `OnCalendar=Mon 03:00`); `install.sh` enables it via `systemctl --user` alongside the rootless `podman.socket`.

The script exits 0 immediately when invoked from a non-interactive shell (stdin is not a TTY) without cached sudo credentials. `install.sh` calls it during bootstrap; under `./install.sh` stdin is a TTY and sudo can prompt, but under agent or CI runs there is no TTY and an uncached sudo would hang or fail. Skipping cleanly lets the rest of the bootstrap finish. The user re-runs `bash scripts/linux/install-linux-system-config.sh` manually afterwards. After the file-install loop, the same script iterates an explicit `REMOVED_ETC_PATHS=(…)` array hard-coded near the top of the cleanup section and `rm -f`s each entry that still exists (including dangling symlinks, via `[[ -e || -L ]]`). When deleting a tracked file under `system/linux/etc/`, add its absolute `/etc` path to that array in the same commit — that is the manifest every machine reads to clean up orphans after pulling the deletion. The earlier `git status`-based detector was removed because it only saw uncommitted changes (clean worktree after pull == orphan persists); `git log --diff-filter=D` was rejected for the opposite reason (no idempotency, would retry removal forever). Entries can stay in the array for a release cycle or two with no cost — `rm -f` is a no-op once the path is gone. Then it runs `systemctl daemon-reload` (when `system/linux/etc/systemd/system/` exists; this is a no-op now that the system-scope container-prune units have moved to user scope, but it stays as a guard for any future system-scope unit drop-in), `loginctl enable-linger "${SUDO_USER:-$USER}"` (so the user-scope `podman.socket` and `podman-prune.timer` keep running across logouts when rootless is the supported mode), and `systemctl mask podman.socket podman.service` at the system (rootful) layer (rootless-enforcement: blocks the system-wide rootful API socket only; the user-scope rootless socket is unaffected). The orphaned system-scope container-prune unit paths are listed in `REMOVED_ETC_PATHS` so a clean removal of the old Docker packages no longer leaves stale unit files behind. It then configures firewalld for Tailscale and VMware: IPv4 masquerade on the default zone (`firewall-cmd --permanent --add-masquerade`), `tailscale0` bound to the `trusted` zone (`--zone=trusted --add-interface=tailscale0`), and UDP 41641 (WireGuard) + UDP 3478 (STUN) opened on the `public` zone (`--zone=public --add-port=…/udp`). Each step is gated on its own `--query-*` probe and a single `--reload` runs at the end only if anything changed. It then sweeps `/etc/NetworkManager/conf.d/` for dangling symlinks (left behind when Home Manager symlinks into `/nix/store/` outlive the store path during Nix decommission) and removes only those — live symlinks and regular files are left alone, so the sweep can't accidentally remove a drop-in placed by another tool. Finally, it runs `systemctl reload NetworkManager` (gated on `systemctl is-active --quiet NetworkManager`, so hosts using systemd-networkd or no NetworkManager are skipped) to pick up the cleaned state and any freshly-installed `/etc/NetworkManager/conf.d/` drop-ins without restarting the service or disrupting active connections. The Tailscale exit-node and VMware NAT egress paths source-NAT through the host's primary interface, which lives in the default zone on Fedora Workstation, and they ride on the IPv4/IPv6 forwarding the `sysctl.d/` drop-in enables. The step is idempotent (`--permanent --query-masquerade` before mutating) and gated on `firewall-cmd --state` so it skips cleanly when firewalld is masked, missing, or replaced by another backend.

### Runtime agent config

- `agents/skills/` is the runtime skill tree and `agents/.skill-lock.json` its lockfile — both managed by the `skills` CLI (`npx skills`) and tracked in the repo. `scripts/bootstrap/link-repo-trees.sh` links `agents/skills` to `~/.agents/skills` (OpenCode) and `~/.claude/skills` (Claude Code), and `agents/.skill-lock.json` to `~/.agents/.skill-lock.json`. The lockfile's `lastSelectedAgents` array lists every agent `npx skills` distributes to (it already includes `codex`, `cursor`, `gemini-cli`, and others), so re-running `npx skills` also syncs the selected skills into those tools' own skill dirs — e.g. Codex picks them up under `~/.codex/skills` (alongside its built-in `~/.codex/skills/.system` skills, which are left intact). `agents/skills/` holds a mix of CLI-managed and hand-authored skills: you MAY add or edit a skill by hand (create `agents/skills/<name>/SKILL.md` directly), but **check the source before editing an existing one** — a skill tracked in `.skill-lock.json` (installed via `npx skills`) or the `glab` skill (`glab skills install`) is CLI-managed and a hand edit is overwritten on the next CLI run. `agents/.skill-lock.json` itself is CLI-owned — don't hand-edit it. Codex also reads this same tree directly: it scans the **USER**-scope `$HOME/.agents/skills` and [follows symlinked skill folders](https://developers.openai.com/codex/skills#where-to-save-skills), so the existing `~/.agents/skills → agents/skills` symlink exposes **every** skill here to Codex — hand-authored (e.g. the rule-skills `git-workflow`, `pr-mr`, `gitlab-issues`, `ci-cd-monitoring`, `js-package-managers`) and CLI-managed alike — with no per-skill links and no `~/.codex/skills` wiring needed. (The `npx skills` copy into `~/.codex/skills` mentioned above is thus redundant for discovery; it is harmless and left intact.) A new hand-authored skill is visible to all three tools the moment its `agents/skills/<name>/SKILL.md` exists — no `link-repo-trees.sh` change.
- The `glab` skill is installed separately by the bootstrap step `mise exec glab -- glab skills install -f --path ./agents/skills`, so it is **not** tracked in `agents/.skill-lock.json` (the `skills` CLI owns that lockfile for the skills it manages, e.g. `find-skills`, `playwright-cli`).
- An `AGENTS.md` MUST NOT exist under `agents/skills/`. That subtree is linked into `~/.agents/skills` and `~/.claude/skills`, so a file there could be injected into every agent run from this user account. Put repo guidance in `agents/AGENTS.md` (a sibling of `skills/`, not linked into any tool) instead.
- `agents/SHARED_AGENTS.md` is the cross-tool agent rules file. `scripts/bootstrap/link-repo-trees.sh` symlinks it to each AI tool's global rules path: `~/.config/opencode/AGENTS.md` (OpenCode), `~/.codex/AGENTS.md` (Codex), and `~/.claude/CLAUDE.md` (Claude Code). Edit `agents/SHARED_AGENTS.md` once; every linked tool sees the change. Add support for a new tool by adding a new entry in `scripts/bootstrap/link-repo-trees.sh` and updating the linkage table in [`agents/README.md`](agents/README.md).
- `agents/commands/` is the cross-tool shared slash-command / prompt directory. `scripts/bootstrap/link-repo-trees.sh` symlinks the whole directory into each tool's command path: `~/.config/opencode/commands` (OpenCode slash commands) and `~/.codex/prompts` (Codex prompts). Each `*.md` file is a single command/prompt and works in either tool — keep the body tool-agnostic. Add a new command by dropping a `<name>.md` file in `agents/commands/`; no further wiring is needed. Add support for a new tool by adding a new entry in `scripts/bootstrap/link-repo-trees.sh` pointing that tool's command path at `agents/commands` and updating the linkage table in [`agents/README.md`](agents/README.md).
- `home/dot_config/opencode/AGENTS.md` MUST NOT exist. `~/.config/opencode/AGENTS.md` is already an explicit symlink to `agents/SHARED_AGENTS.md` (created by `link-repo-trees.sh`); a sibling source under `home/dot_config/opencode/` would conflict with that link. Put cross-tool rules in `agents/SHARED_AGENTS.md`. Put OpenCode-only rules in a new file under `agents/` linked separately by `link-repo-trees.sh`.
- `home/.config/opencode/commands/` MUST NOT exist. `~/.config/opencode/commands` is already an explicit symlink to `agents/commands`; a sibling source under `home/.config/opencode/commands/` would conflict with that link. Put new slash commands in `agents/commands/`. The OpenCode glob link for `home/dot_config/opencode/` is intentionally narrowed to `*.{json,jsonc}` so a stray `commands/` subdir there cannot collide.
- Keep all agent rule files in sync when changing agent workflow rules. In this repo that currently means this file, `home/AGENTS.md`, `agents/AGENTS.md`, and `agents/SHARED_AGENTS.md`. The shared file is loaded globally by OpenCode, Codex, and Claude Code via the symlinks above, so changes propagate to all three tools immediately.
- **Codex `config.toml` is rendered, NOT symlinked.** Unlike `~/.codex/AGENTS.md` and `~/.codex/prompts` (whole-dir symlinks into `agents/` created by `link-repo-trees.sh`), `~/.codex/config.toml` MUST NOT be a symlink: Codex writes machine-local state back into that same file — most importantly per-project trust decisions under `[projects."<path>"] trust_level` — and has no `include`/import directive to merge a separate tracked file in (verified against the `openai/codex` config loader). Symlinking it would let Codex mutate the repo on every "trust this folder?" prompt. Instead the shared Codex settings live in [`codex/codex-config.managed.toml`](codex/codex-config.managed.toml) (a tracked "simple TOML" file — one single-line `key = value` per line under standard `[table]` headers; no `[projects]`), and a bootstrap step (`scripts/bootstrap/configure-codex-config.mjs`, called by `install.sh` / `install.ps1`, with `.sh`/`.ps1` wrappers for manual runs) merges only those declared keys into `$CODEX_HOME/config.toml` (default `~/.codex/config.toml`), updating/inserting them with targeted TOML-safe edits and preserving every other byte (the `[projects]` trust table, `preferred_auth_method`, anything you added). It is idempotent, backs up to `config.toml.bak` before changing, aborts without writing rather than corrupt an ambiguous multi-line value, and is a no-op when the managed file has no active keys (the default). There is no default-profile mechanism in current Codex (the top-level `profile` key is legacy and a profile only activates via `--profile`), and the system layer (`/etc/codex/config.toml`) is root-owned — the renderer is the simplest always-applied, no-root, cross-platform path. Editing a value in the managed file re-points that key on the next run; **deleting** a line there does NOT remove the key from the live file (the renderer only adds/updates), so drop unwanted keys from `~/.codex/config.toml` by hand. `home/.codex/` does not (and should not) exist — the managed source lives in the top-level [`codex/`](codex/) dir. **`~/.codex/hooks.json` is the opposite case:** Codex never writes back to its hooks file, so [`codex/hooks.json`](codex/hooks.json) (Codex lifecycle hooks — see the haptic section) IS a plain symlink, linked by `install.sh` on Linux + macOS bootstrap steps. So `codex/` mixes one rendered file (`config.toml`) and one symlinked file (`hooks.json`); the rule of thumb is whether Codex mutates the target.

### TypeScript/JS packages — Yarn Berry monorepo at `packages/`

- **All TypeScript/JavaScript libraries live under [`packages/`](packages/)** as members of a single Yarn Berry workspace. The workspace ROOT is `packages/` itself — `packages/package.json` is the private `@h82/dotfiles` root (`"workspaces": ["*"]`). The monorepo root is deliberately `packages/`, **NOT** the repo root: there is no `package.json`/`yarn.lock`/`.yarn/` at `/` so `cd ~/dotfiles` stays a plain checkout, while `cd ~/dotfiles/packages` is the Yarn project. Do **not** add a root-level `package.json`.
- **Add a new TS library** by creating `packages/<name>/` with its own `package.json` (`@h82/<name>`, `private: true`, `"type": "module"`). The `"workspaces": ["*"]` glob picks it up automatically. Reference sibling packages with the `workspace:*` protocol. Build members with [`tsdown`](https://tsdown.dev) (Rolldown-based; config in `<name>/tsdown.config.ts`, `fixedExtension: false` so ESM output is `.js`/`.d.ts` matching the `exports` map) and type-check separately with `tsc --noEmit`.
- **One lockfile**: the workspace-root `packages/yarn.lock` is the only lockfile and is **tracked**. `node_modules/` hoists to `packages/node_modules/`. There are no per-member lockfiles or `node_modules`.
- **Yarn config**: `packages/.yarnrc.yml` sets only `nodeLinker: node-modules` (so plain `tsc`/`node --test` resolve normally). All hardening from the user-global `~/.yarnrc.yml` — `enableScripts: false`, `enableImmutableInstalls: true`, exact pins (`defaultSemverRangePrefix: ""`), the 1-week cooldown gate (`npmMinimalAgeGate`) — cascades and MUST NOT be relaxed. Pin every dependency to an exact, cooldown-valid (≥7 days old) version.
- **First install gotcha**: `enableImmutableInstalls: true` makes the very first `yarn install` (which must create `packages/yarn.lock`) fail. Use `yarn add <pkg>@<exact>` (mutating) or `yarn install --no-immutable` for the initial lockfile; thereafter `yarn install --immutable` is clean.
- **Task runner: Turborepo.** `packages/turbo.json` defines the `build`, `typecheck`, `test`, `lint`, `format`, and `format:check` tasks (`build` outputs `dist/**`; `typecheck`/`test` `dependsOn` `^build`; `lint`/`format`/`format:check` have no deps since ESLint/Prettier read source directly; `format` is `cache: false` because it writes). The workspace-root scripts (`yarn build`/`yarn test`/`yarn typecheck`/`yarn lint`/`yarn format`/`yarn format:check`) delegate to `turbo run …`, so run those from `packages/`. Each member exposes matching scripts that turbo invokes. `turbo` is pinned exact in the root `devDependencies` and has no install lifecycle script (safe under `enableScripts: false`). Turbo's caches (`packages/.turbo/`, `packages/*/.turbo/`) are git-ignored — keep them ignored or turbo folds its own logs into the input hash and never caches.

## Common mistakes

- Installing chezmoi via `brew`, `dnf`, `apt`, `winget`, `go install`, or any other method. **Always use mise-managed `mise exec chezmoi@2.70.5 -- chezmoi ...`** — ephemeral, exact version, no global install.
- Using an ambient `chezmoi` on `$PATH` — it may be a different version than the pinned `2.70.5`.
- Pointing chezmoi at the repo root instead of `home/`. The source root is `home/` (`.chezmoiroot = home`); always pass `--source "$REPO_ROOT/home"` or rely on `.chezmoiroot` discovery.
- Renaming source files under `home/` without understanding chezmoi's prefix encoding. `dot_` replaces a leading `.`; `private_` sets mode `0600`/`0700`; `executable_` sets the execute bit. A wrong prefix silently changes the target path or permissions.
- Editing `$HOME` target files directly. chezmoi will overwrite them on the next apply. Edit the source under `home/` instead.
- Adding live-symlink paths (e.g. `agents/`, `.config/opencode/commands`, `.config/opencode/AGENTS.md`) to the chezmoi source tree. Those are created by `scripts/bootstrap/link-repo-trees.sh` and listed in `.chezmoiignore.tmpl` — chezmoi must not manage them.
- Committing SSH/age/GPG private keys, API tokens, `.env` files, or anything else `.gitignore` is meant to keep out.
- `cp` for `/etc/` files, or `sudo cp` instead of `sudo install -D -m <mode>`.
- Adding `.sh` without matching `.ps1`, or vice versa.
- Adding a top-level directory without a `README.md`, or moving things without updating the layout block in this file.
- Adding an `AGENTS.md` under `agents/skills/`. That subtree links into `~/.agents/skills` and `~/.claude/skills` and could be injected into every agent run; put repo guidance in `agents/AGENTS.md` instead.
- Adding `home/.config/opencode/AGENTS.md`. `~/.config/opencode/AGENTS.md` is already an explicit symlink to `agents/SHARED_AGENTS.md`; a second source would conflict. Edit the shared file, or add a new tool-specific file under `agents/` with its own `link:` entry.
- Adding `home/.config/opencode/commands/`. `~/.config/opencode/commands` is already an explicit symlink to `agents/commands`; a sibling source would conflict. Put new slash commands in `agents/commands/`.
- Hand-editing `~/.config/opencode/AGENTS.md`, `~/.codex/AGENTS.md`, `~/.config/opencode/commands/*`, or `~/.codex/prompts/*`. Those are symlinks into `agents/` — edit the source under `agents/`.
- Symlinking `~/.codex/config.toml` into the repo (or adding `home/.codex/config.toml`). Codex writes machine-local `[projects]` trust state back into that file and has no `include` directive, so a symlink lets Codex mutate the repo. Put shared keys in `codex/codex-config.managed.toml`; the `configure-codex-config` renderer merges them while preserving the machine-local rest.
- Updating agent workflow rules in only one `AGENTS.md` when the change also applies to the linked runtime agent docs.
- Adding a `README.md` under `.github/`. It would shadow the root `README.md` on GitHub's landing page — document `.github/` in `.github/WORKFLOWS.md` instead.

## Solaar haptic playback (MX Master 4)

The MX Master 4 exposes HID++ feature `0x19B0` (HAPTIC) with 16 built-in waveforms. Solaar surfaces this as the `haptic-play` setting — a write-only action setting, not stateful. From shell:

```bash
solaar config "MX Master 4" haptic-play "<WAVEFORM>" 2>/dev/null || true
```

The `2>/dev/null || true` swallows the Solaar 1.1.19 + Python 3.14 `Gio.Application.run` marshalling crash (`TypeError: Unable to marshal str as an array`) — the same upstream bug that breaks `solaar config divert-keys`. The runtime HID++ command is dispatched before the persistence step crashes, so the mouse pulses regardless; only the post-dispatch persistence layer crashes. Track the upstream fix at <https://github.com/pwr-Solaar/Solaar>.

**Spawn cost per pulse is ~3.7 SECONDS** on this host (cold-cache `time` measurement, repeated three times) — the CLI bootstraps the full Solaar Python stack on every call. That makes `solaar config haptic-play` **unsuitable for any latency-sensitive feedback**: if you fire it from inside a Solaar rule via `Execute`, the pulse arrives 3-4 seconds after the trigger, which is well past any normal button hold.

**Inside Solaar rules, use the [`mxm4-haptic`](crates/mxm4-haptic/) client** — `Execute: [mxm4-haptic, "<WAVEFORM>"]`. The [`mxm4-haptic`](crates/mxm4-haptic/) crate builds **three** Rust binaries into `~/.local/bin/` from a single `cargo install` during bootstrap. It runs on **Linux + macOS + Windows**: the daemon reaches the device through the cross-platform [`hidapi`](https://crates.io/crates/hidapi) crate (Linux **hidraw** backend, macOS **IOKit** shared open, Windows pure-Rust **`windows-native`** backend), pinned `=2.6.5` with `default-features = false` + `["linux-static-hidraw", "macos-shared-device", "windows-native"]`. The daemon links two confined, justified exceptions to the crates "zero-dep" rule: `hidapi` (no pure-`std` path to HID off Linux) on every platform, and `windows-sys` (no AF_UNIX in std on Windows, so the IPC server is a Win32 named pipe via `CreateNamedPipeW`) gated to `[target.'cfg(windows)'.dependencies]`. The Linux build needs `libudev` headers (`systemd-devel`, on macOS the system IOKit frameworks suffice; the Windows backend is pure Rust, no C toolchain). Both the Linux and macOS bootstraps build and autostart the daemon (`systemd --user` on Linux, a launchd agent on macOS) — see the macOS note at the end of this section. **Windows is code-level compatible only** (verified via `cargo check --target x86_64-pc-windows-gnu`): bootstrap does not build, install, or autostart it there, so build and run it manually; the notify bridge stays a Linux-only stub, and Logitech Options+ may need closing if HID writes fail.

- **`mxm4-hapticd`** — a `systemd --user` daemon ([`mxm4-hapticd.service`](home/.config/systemd/user/mxm4-hapticd.service)) that is the **sole owner** of the Bolt receiver's HID++ session. It discovers the MX Master 4 and its HAPTIC feature index **over `hidapi`** (enumerating by Bolt VID/PID `046D:C548` and the HID++ control interface — USB interface 2 on Linux, vendor usage page `0xFF00` on macOS and Windows where `interface_number` is `-1`) — enabling receiver `0x41` connection notifications, re-announcing to learn which slots are connected, then sending Root `GetFeature(0x19B0)` to each connected slot; the slot that answers with a feature index instead of `ERR_INVALID_FEATURE_INDEX` is the mouse. **No `solaar show`, no on-disk cache** — device state lives only in memory and is re-discovered on reconnect (it watches `0x41` for disconnect/reconnect, including a different receiver slot). A **single I/O-owner thread** holds the one `HidDevice` (that type is `Send` but not `Sync`, and macOS IOKit does not promise a second handle sees the same unsolicited reports Linux hidraw broadcasts to every fd): it polls `read_timeout()` to drain input reports and, between reads, serializes playback with debounce + per-pulse pacing, writing the HID++ Long play report (feature `0x19B0`, function 4, ~1-2 ms, no ack wait). A separate accept thread feeds it waveform ids over an `mpsc` channel.
- **`mxm4-haptic`** — the thin one-shot client spawned by Solaar rules; validates the waveform name and forwards it to the daemon over the IPC endpoint (AF_UNIX socket on Unix — `$XDG_RUNTIME_DIR/mxm4-haptic.sock` on Linux; `$TMPDIR/mxm4-haptic.sock` on macOS, which has no `XDG_RUNTIME_DIR`; the `\\.\pipe\mxm4-haptic` named pipe on Windows, opened via std's `CreateFileW`-backed `OpenOptions`), then exits. No device access (does not link `hidapi`) and no fallback — the daemon is the only writer, so if it is down the pulse is simply skipped.
- **`mxm4-haptic-notify`** — a `systemd --user` notification bridge ([`mxm4-haptic-notify.service`](home/.config/systemd/user/mxm4-haptic-notify.service)) that eavesdrops `org.freedesktop.Notifications.Notify` via `dbus-monitor` and forwards an urgency-mapped waveform to the daemon (critical → `SHARP COLLISION`, normal → `SUBTLE COLLISION`, low ignored; tunable via the `MXM4_NOTIFY_*` env vars in the unit). **Linux-only** — there is no public API to eavesdrop other apps' notifications on macOS, so off Linux this binary compiles to a stub that exits with a message.

Rejected alternatives (do not reintroduce): `solaar config haptic-play "<WAVEFORM>"` spawns the full Solaar CLI (~3.7 s cold — the pulse lands seconds after the trigger); `Set: [null, haptic-play, "<WAVEFORM>"]` is in-process but waits for an HID++ ack (`PlayHapticWaveForm` doesn't set `no_reply=True` on `FeatureRW`), and Bolt-wireless ack variance (30-300 ms) makes pulses feel "missing".

There is **no `~/.cache/mxm4-haptic.json`** any more — discovery is over `hidapi` and in-memory. The HID++ discovery wire format (Root `GetFeature`, the `0x41` connection notification, the receiver register writes that enable notifications / re-announce devices, and the request `sw_id 0x0E` chosen to avoid colliding with Solaar's fixed `0x0B`) is documented in [`crates/mxm4-haptic/src/lib.rs`](crates/mxm4-haptic/src/lib.rs) and the daemon source, verified against libratbag and Solaar. Coexistence with the running Solaar autostart session is safe on Linux: the daemon opens the **hidraw** backend (not libusb, which would detach the kernel driver), so it reads the same broadcast reports Solaar does and its receiver-register writes are idempotent.

chezmoi applies the unit files, and `install.sh` enables both `mxm4-hapticd.service` and `mxm4-haptic-notify.service` automatically in a guarded step right after the `cargo install` build (it soft-skips and exits 0 when no user manager bus is reachable — agent / CI / chroot runs). There is no `--now`: plain `enable` only wires the `WantedBy` symlinks, so systemd starts each unit when its target is reached — `mxm4-hapticd.service` with `default.target`, and `mxm4-haptic-notify.service` when `graphical-session.target` comes up. If the auto-enable was skipped (no session bus during bootstrap), enable both manually once a session exists: `systemctl --user daemon-reload && systemctl --user enable mxm4-hapticd.service mxm4-haptic-notify.service`.

**macOS autostart.** `install.sh` builds the daemon + client with `cargo install … --bin mxm4-hapticd --bin mxm4-haptic` (the notify bridge is Linux-only, skipped), links the launchd agent [`home/Library/LaunchAgents/dev.h82.mxm4-hapticd.plist`](home/Library/LaunchAgents/dev.h82.mxm4-hapticd.plist) into `~/Library/LaunchAgents/`, and `launchctl bootstrap`s it — the macOS counterpart of the Linux `systemctl --user enable`. The plist's `RunAtLoad` + `KeepAlive` mirror the unit's `WantedBy=default.target` + `Restart=always`; `exec "$HOME/.local/bin/mxm4-hapticd"` runs through `/bin/sh` because launchd does not expand `$HOME` in `ProgramArguments`, and the daemon resolves its socket under the per-user `$TMPDIR` that launchd sets for GUI agents. The load step soft-skips without a GUI session (`gui/$(id -u)` absent in SSH/CI). macOS has no Solaar to fire the client from rules, so the natural driver there is the AF_UNIX socket — e.g. the [`@h82/opencode-mxm4-haptic`](packages/opencode-mxm4-haptic/) plugin or the [`@h82/mxm4-haptic`](packages/mxm4-haptic/) TS client. See [`home/Library/LaunchAgents/README.md`](home/Library/LaunchAgents/README.md).

The `mxm4-haptic` client also emits its CLI spec via `--usage` ([usage](https://usage.jdx.dev) KDL, waveform `choices` generated from the `WAVEFORMS` table so they never drift). The Linux bootstrap pipes it through `usage` into a static zsh completion at `~/.config/zsh/completions/_mxm4-haptic` (on `fpath` via [`home/.config/zsh/.zprofile`](home/.config/zsh/.zprofile), so it loads before Prezto's `compinit`) — giving `mxm4-haptic <TAB>` waveform-name completion. The completion is a generated artifact, not tracked.

Waveforms (from `logitech_receiver.hidpp20_constants.HapticWaveForms`):

| Theme | Names |
|---|---|
| State / collision | `SHARP STATE CHANGE`, `DAMP STATE CHANGE`, `SHARP COLLISION`, `DAMP COLLISION`, `SUBTLE COLLISION`, `WHISPER COLLISION` |
| Alerts | `HAPPY ALERT`, `ANGRY ALERT`, `COMPLETED`, `MAD` |
| Rhythmic | `SQUARE`, `WAVE`, `FIREWORK`, `KNOCK`, `JINGLE`, `RINGING` |

Multi-word names **require quoting**. Intensity scales with `haptic-level` (0-100, persists across reconnect; set in the Solaar GUI — the dotfiles bootstrap deliberately does NOT enforce a level since the helper's no-ack-wait path makes mid-range intensities reliably perceived). The thumb-button rules in [`home/.config/solaar/rules.yaml`](home/.config/solaar/rules.yaml) consume this API: each of the four DIRECTIONAL mouse-gesture rules (Up/Down/Left/Right) fires `Execute: [mxm4-haptic, "SUBTLE COLLISION"]` after invoking its KWin shortcut, giving a Logi-action-ring-style tactile confirm that the flick registered (the press-with-no-movement KRunner rule is left silent — it is self-confirming). An earlier Haptic-hold-detection rule there once pulsed a waveform from a `Later` callback when hold-mode was detected; that stateful pattern is gone, replaced by the stateless per-gesture pulse. The live consumers of the daemon are: those Solaar gesture rules, the zsh long-command hook in [`home/.config/zsh/.zshrc`](home/.config/zsh/.zshrc) (fires `JINGLE` when a foreground command exceeds 30 s, skipping Ctrl+C-interrupted jobs, and only when you have looked away — `kdotool` reads KWin's active window and stays silent while a Konsole window is focused), the [`@h82/mxm4-haptic`](packages/mxm4-haptic/) TS client, the [`@h82/opencode-mxm4-haptic`](packages/opencode-mxm4-haptic/) plugin, and the OpenAI Codex lifecycle hooks in [`codex/hooks.json`](codex/hooks.json) (symlinked to `~/.codex/hooks.json` on Linux + macOS; `Stop` → `COMPLETED`, `PermissionRequest` → `RINGING`, the Codex counterpart of the OpenCode plugin — non-managed command hooks must be trusted once via `/hooks`).

For a non-Solaar path (direct HID++ over the Bolt receiver's `:1.2` hidraw to feature `0x19B0`), the byte sequence is what `mxm4-hapticd` implements — see [`crates/mxm4-haptic/src/lib.rs`](crates/mxm4-haptic/src/lib.rs) for the exact packet layout. The full HID++ 2.0 spec documents the rest.

For Node/Bun consumers there is a TypeScript client library, [`@h82/mxm4-haptic`](packages/mxm4-haptic/), a member of the [`packages/`](packages/) Yarn workspace (the `@h82/dotfiles` monorepo rooted at `packages/`). It is a thin client that mirrors the Rust crate's portable surface, exporting only the `WAVEFORMS` catalogue, the derived `WaveformName` type, and a type-safe, flush-confirmed `sendCommand(name: WaveformName)` (waveform lookups and the socket-path resolver are kept internal); it talks to the running `mxm4-hapticd` daemon over the same IPC endpoint (the `$XDG_RUNTIME_DIR/mxm4-haptic.sock` AF_UNIX socket on Unix, the `\\.\pipe\mxm4-haptic` named pipe on Windows) — it does **not** touch hidraw or replace the daemon. It is **not** installed or symlinked by chezmoi (a library ships nothing to `~/.local/bin`); the Linux bootstrap only `yarn build`s the workspace in place. The Rust `mxm4-haptic` client binary remains the latency-critical path used by Solaar rules.

One consumer of `@h82/mxm4-haptic` is [`@h82/opencode-mxm4-haptic`](packages/opencode-mxm4-haptic/), an OpenCode plugin (also a `packages/` workspace member) that pulses a waveform on OpenCode events: `session.idle` → `COMPLETED` (a tactile "agent finished" buzz, gated so only a fully-idle root session — not each sub-agent — fires it), `session.error` → `MAD`, `permission.updated` → `RINGING`, and the `Question` tool call → `RINGING` (caught via the `tool.execute.before` hook, since it emits no `permission.updated`).
